# 00. Overview

## Что строим

Систему-ассистент 1-й линии поддержки, работающую локально, использующую внутренний LLM (GigaChat) и индексирующую корпоративные знания (база знаний + история закрытых тикетов).

Система состоит из четырёх взаимосвязанных модулей:

1. **Ingest pipeline** — конвейер обработки тикетов: чтение CSV-выгрузки → нормализация → PII-маскирование → классификация резолюции → генерация выжимки → дедупликация → индексация.
2. **Retrieval + RAG-assistant** — гибридный поиск по индексу + сборка промпта + ответ LLM с цитированием источников.
3. **Categorizer** — автоматическая категоризация и приоритизация входящего обращения.
4. **Evals** — eval-набор и runner для контроля качества (faithfulness, helpfulness, retrieval recall).

Плюс standalone HTML-приложение для визуализации и операторской работы с системой.

## Целевые сценарии

### Сценарий 1: Поддержка получает обращение

Оператор открывает новое обращение в нашем UI (или в будущем — в Service Manager с виджетом). Запускает «Проанализировать тикет». Ассистент:

1. Автоматически категоризирует обращение (модуль, тип, срочность).
2. Ищет в индексе похожие закрытые тикеты, статьи KB, плейбуки.
3. Формирует ответ: возможная причина, шаги диагностики, готовый драфт ответа клиенту, ссылки на источники.
4. Оператор проверяет, правит, применяет.

### Сценарий 2: Загрузка истории тикетов

Сотрудник 2-й линии или администратор кладёт CSV-выгрузку из Service Manager в систему. Запускает ингест. Система:

1. Парсит и нормализует.
2. Маскирует PII (ФИО, номера заявок, суммы).
3. Через LLM классифицирует, какие тикеты реально решены.
4. Для решённых — генерирует структурированную выжимку (симптом, причина, решение).
5. Дедуплицирует похожие.
6. Индексирует в векторную БД и полнотекстовый индекс.

После этого знания доступны RAG-ассистенту.

### Сценарий 3: Контроль качества

Перед изменением промпта или модели команда запускает eval-набор. Прогон:

1. Берёт 100–200 эталонных кейсов.
2. На каждом запускает retrieval + assistant.
3. Считает Recall@5, faithfulness (через LLM-судью), helpfulness (через LLM-судью), adversarial pass rate.
4. Сравнивает с предыдущим прогоном — нет ли регрессии.

Без зелёного прогона evals — изменение не катится.

## Цели

- **Снизить нагрузку на 1-ю линию.** Цель — 30% времени на типовой тикет за счёт автоматизации поиска решений.
- **Дать прозрачность.** Категории, тренды, метрики на дашборде.
- **Сделать опыт переиспользуемым.** Закрытый нетипичный тикет → завтра знания доступны всей команде.
- **Безопасность данных.** Никакой PII в LLM-запросах и в индексе. Никаких внешних API без явного разрешения.

## Не-цели (явно вне скоупа)

- Не пытаемся заменить человека-оператора. Ассистент всегда draft, человек проверяет.
- Не отвечаем клиентам автономно. Никогда. Ответ оператора всегда проверяется.
- Не принимаем бизнес-решения по заявкам клиентов. Не оцениваем кредитоспособность. Не одобряем/отклоняем.
- Не интегрируемся с реальным Service Manager в MVP. На старте — только CSV.
- Не делаем встроенный виджет для SM в MVP. Только отдельное web-приложение.
- Не делаем многопользовательскую авторизацию в MVP. Простой `X-User-Id` для аудита.
- Не реализуем real-time коллаборацию между операторами.

## Метрики успеха MVP

После пилотного запуска (3 месяца) проверяем:

| Метрика | Цель |
|---|---|
| Adoption (тикеты с использованием ассистента / все тикеты) | > 50% |
| Helpful rate (👍 / (👍 + 👎)) | > 70% |
| Faithfulness на eval-наборе | > 0.95 |
| Retrieval Recall@5 на eval-наборе | > 0.85 |
| Adversarial pass rate (защита от prompt injection) | 100% |
| Среднее время от открытия тикета до отправки ответа | -25% к baseline |

## Размер и сложность

Реалистичная оценка: **MVP реализуется за 4–6 недель** силами 1 разработчика с использованием Claude Code в качестве основного кодогенератора. Минимально жизнеспособная версия (без UI, только API + ингест + RAG) — 2–3 недели.

Объём кода — ориентировочно 5–8 тыс. строк Python + 1.5–2 тыс. строк JS/HTML/CSS. Это не индустриальный продукт, а внутренний инструмент с возможностью масштабирования.
