0. Стенд только подняли — что увидишь

После ./run.sh на /ui/#/dashboard видишь сводку. Если БД пустая, KPI будут нулевыми. Это нормально.

  • Слева — сайдбар с разделами. Сейчас читаешь страницу «Инструкция».
  • В правом верхнем углу — переключатель тёмной/светлой темы.
  • В «Здоровье» можно проверить, что ассистент достучался до LLM и эмбеддингов.

1. Загрузить тикеты в индекс

Открой «Ингест». Это страница, через которую данные попадают в индекс ассистента.

  1. Если у тебя есть CSV (формат — см. вкладку «Артефакты» → артефакт «Исторические закрытые тикеты»), перетащи файл в дроп-зону.
  2. Нажми «Запустить ингест». Ниже появится job с прогресс-баром (обновляется по SSE).
  3. Когда job станет succeeded, открой «Тикеты» — данные уже в БД и в поиске.

Демо-CSV (200 синтетических тикетов) лежит рядом с проектом в data/sample_tickets.csv и сидится автоматически при первом ./run.sh.

2. Завести базу знаний

База знаний (KB) — это инструкции и SOP, на которые ассистент опирается, когда тикетов мало или решение неочевидное.

  1. Открой «База знаний».
  2. Нажми «+ Новая статья» — заполни тему, модуль, тело в markdown. Сохрани — статья режется на чанки и попадает в индекс автоматически.
  3. Либо «Импорт zip / md»: zip-архив с *.md-файлами, для bulk-загрузки.

Формат и пример — в «Артефакты» → «Статьи базы знаний».

3. Спросить ассистента

  1. Открой «Ассистент».
  2. Введи вопрос в стиле первой линии поддержки: «Не загружается скан паспорта у клиента, файл 12 МБ».
  3. Опционально: над полем ввода — поиск тикетов. Найди и «прицепи» тикет — ассистент учтёт его контекст.
  4. Если хочешь, чтобы ассистент мог задать уточняющий вопрос вместо ответа — поставь галку «Разрешать уточняющие вопросы».
  5. Нажми «Спросить». Сначала появятся источники, потом потоковый ответ. Цифры [1], [2] в тексте — ссылки на источники справа. Наведи на цифру — карточка подсветится.

4. Оценить ответ и (если хороший) — в eval-набор

Под каждым ответом ассистента есть три кнопки:

  • 👍 Помогло / 👎 Не помогло — сохраняется в БД. Эти оценки потом видны на «Слабых ответах».
  • + В eval-набор — открывает диалог: query, expected_sources, must_mention/must_not, edge_case_type. Сохраняется как JSON-кейс в evals/cases/ — потом этот кейс прогоняется при каждом запуске evals.
  • ★ В few-shot — отправляет пару (вопрос, ответ) на модерацию. После approve в «Few-shot» попадает в системный промпт следующих вызовов.

5. Разобрать новый тикет — категоризация + RAG + драфт

  1. Открой «Тикеты», кликни на строку.
  2. В детальной карточке справа — кнопка «Анализ». Жмёшь — за один вызов получаешь:
    • Категоризацию (модуль, тип, срочность, на какую группу эскалировать);
    • Ответ ассистента со ссылками на источники;
    • Готовый драфт ответа клиенту.
  3. Скопировать в свой ответ — обычное Cmd+C.
  4. Если тикет добавился позже и не в индексе — «Переиндексировать» рядом.

6. Прогнать evals

Evals — это «контрольная работа» для ассистента. Набор эталонных кейсов прогоняется через текущий промпт+LLM+индекс, считаются метрики (faithfulness, helpfulness, recall@5, MRR), результат сравнивается с предыдущим прогоном.

  1. Открой «Evals».
  2. Кнопка «Запустить прогон» — берёт все кейсы из evals/cases/.
  3. Через минуту-другую прогон появится в таблице. Кликни — увидишь per-case карточки: где ассистент проседает.
  4. В блоке «Сравнение прогонов» выбери два прогона (A и B) — увидишь, какие кейсы улучшились/просели, и Δ агрегатов.

7. Чинить «слабые ответы»

Все ответы, где оператор поставил 👎, ассистент не нашёл источников или явно отказался, копятся на «Слабые ответы».

  1. Открой страницу, фильтруй по периоду и причине.
  2. По каждой строке: 1 клик «+ В eval-набор» — превращаешь проблему в постоянную проверку.
  3. Если в KB не хватает статьи — заведи её на странице KB. Если ответ должен был быть из тикета, которого нет — догружай через ингест.

8. Поправить системный промпт в sandbox'е

«Промпты» — это playground для безопасной правки.

  1. Слева текущий активный промпт (промпт A). Справа — твой черновик (промпт B).
  2. Под полями выбери eval-кейс или введи произвольный запрос — жми «Сравнить».
  3. Внизу — ответ A vs ответ B. Видно, что меняется.
  4. Когда устраивает — «Сохранить B как версию». Можно «сразу активировать» — следующие ответы ассистента пойдут через новый промпт.
  5. В таблице версий — кнопки активации / удаления неактивных.

9. Зафиксировать «золотой» ответ как few-shot

Few-shot пример — это пара (вопрос, эталонный ответ), которая подмешивается в системный промпт. Помогает ассистенту копировать твой стиль ответа.

  1. Под хорошим ответом нажми ★ В few-shot — пример уйдёт со статусом pending.
  2. Открой «Few-shot», отфильтруй по «pending».
  3. Если ответ действительно эталонный — нажми Approve. Следующий вызов ассистента уже его учтёт.
  4. Можно создать пример руками — кнопка «+ Новый пример».

10. Настроить PII под свои поля

Базовый masker закрывает ФИО, телефоны, e-mail, паспорта. Но у банка часто есть свои форматы: номера договоров, application_id, и т.д.

  1. Открой «PII playground».
  2. Вставь текст с твоими настоящими полями.
  3. В блоке «Дополнительные паттерны» добавь свой regex — например, \bAPP-\d{8,12}\b с меткой APPLICATION_ID.
  4. Жми «Замаскировать» — увидишь, что попало.
  5. Когда regex проверен — закрепи его в проекте (формат описан в «Артефакты» → «Словарь PII / секретов»).

11. Проверить здоровье и деньги

  • «Здоровье» — пинги LLM, embeddings, vector_store, FTS. Coverage по модулям: где KB пустая.
  • «Стоимость» — токены и латентность по типам вызова (answer / summary / judge / categorize). p95-латентность — главное число, на которое смотрит дежурный.
  • «Устаревшее KB» — статьи, которые давно не обновлялись и собирают негативный feedback. Reviewer queue.
  • «Аудит» — лог всех изменений (кто заводил KB, кто запускал ингест, кто менял настройки).

12. Включить алёрты

  1. Открой «Алёрты» — увидишь текущие сигналы и пороги.
  2. Чтобы их получать — заведи Slack/Mattermost incoming webhook и впиши URL.
  3. В «Настройки» → «Алёрты»: включи ALERTS_ENABLED=true, задай ALERTS_WEBHOOK_URL, при желании подкрути пороги p95/no_sources/errors.
  4. Сохрани → перезапусти сервер (Ctrl+C в окне с ./run.sh, потом ./run.sh --no-install).

13. Сменить LLM или включить интеграции

  1. Открой «Настройки».
  2. В разделе «LLM» переключи провайдер (mock / gigachat / yandexgpt / openai_compatible) и впиши креды.
  3. В разделе «Интеграции — TLS / прокси»: галка «Проверять TLS» — выключи, если у вас внутрикорпоративный сертификат, который Python не доверяет.
  4. В разделе «Embeddings» переключи провайдер на local (если есть модель в кэше) или оставь mock для демо.
  5. Жми «Сохранить» — UI запишет изменения в .env. Затем перезапусти uvicorn — новые настройки подхватятся.

Секреты (client_secret, API ключи, webhook'и) в UI отображаются маской ****1234. Чтобы не светить — храни их в .env или менеджере секретов.

Полезные горячие клавиши

  • Ctrl/⌘ + Enter в поле ассистента — отправить.
  • Esc — закрыть модалку.
